# 简历整体结构设计指南

## 核心理念

**"7秒定生死"**：研究表明，面试官平均只花 7.4 秒做初筛决策。简历的信息架构必须让关键价值在极短时间内传递到位。

**"F形扫描"**：面试官的视线轨迹呈F形——先扫顶部，再扫左侧，最后快速浏览中下部。因此：
- 顶部 = 候选人画像（最关键信息）
- 左侧 = 结构清晰的章节标题
- 中下部 = 支撑性细节

## 章节结构（按顺序）

### 第1章：候选人画像（Profile Snapshot）

> 详细设计指南见 `references/profile-snapshot.md`

简历最顶部的"电梯演讲"区域，占据简历顶部约 1/4 的空间。面试官7秒内看到的全部就是这个区域。

必须包含：
- 基础信息行（姓名、年龄、院校、工作年限、当前职位）
- 行业专长与公司履历时间线
- 核心技能标签（6-8个最强技能）
- **一句话定位**（候选人独特价值主张）

### 第2章：核心优势（Key Strengths）

3-5条**量化的**核心成就，快速证明候选人的价值。

**写法要求**：
- 必须量化：用数字说话（人数、百分比、金额、时间）
- 必须是成就而非职责
- 聚焦最近3-5年的成果

**好的例子**：
```
- 主导设计日均处理 2000万+ 请求的分布式交易系统，系统可用性达 99.99%
- 带领 12 人团队从 0 到 1 交付智能风控平台，上线后欺诈率降低 67%
- 推动团队从传统单体架构迁移至微服务，发布效率提升 300%
```

**差的例子**：
```
- 负责后端系统的开发和维护工作（没有量化）
- 参与需求讨论和方案设计（太模糊）
- 有丰富的项目管理经验（空洞）
```

### 第3章：技能矩阵（Skills Matrix）

分类展示候选人的**完整技能集**，让面试官快速了解技术栈广度和深度。

**分类建议**：

```markdown
## 技能矩阵

**编程语言**：Java (精通) · Python (熟练) · Go (熟悉) · TypeScript (熟悉)

**框架 & 中间件**：Spring Boot · Spring Cloud · MyBatis · Redis · Kafka · RocketMQ · Elasticsearch

**数据库**：MySQL · PostgreSQL · MongoDB · TiDB

**云原生 & DevOps**：Kubernetes · Docker · Jenkins · GitLab CI · Prometheus · Grafana

**架构能力**：微服务架构 · DDD · CQRS · 分布式事务 · 高并发设计

**AI & 新技术**：LLM应用开发 · RAG · Prompt Engineering · LangChain
```

**注意事项**：
- 熟练度标注要诚实，面试会验证
- 分类不宜超过6类
- 每类技能控制在一行内（5-8个）
- 把最强、最相关的技能放在前面

### 第4章：工作经历（Work Experience）

**反向时间序列**，从最近的工作开始。每段经历聚焦**成就**而非**职责**。

**格式**：

```markdown
### 公司名称 | 职位 | 起止时间

公司简介（一句话，突出规模/行业地位）

**核心成就**：
- 成就1：[动作] + [量化结果] + [业务影响]
- 成就2：[动作] + [量化结果] + [业务影响]
- 成就3：[动作] + [量化结果] + [业务影响]

**技术栈**：Java · Spring Cloud · Kubernetes · MySQL
```

**写法要点**：
- 每段经历 3-5 条成就，不是罗列日常工作
- 用强动词开头：主导、设计、优化、推动、带领、搭建
- 量化！量化！量化！
- 公司简介让非行业内的面试官也能快速理解背景

### 第5章：代表项目（Key Projects）

选 2-3 个最能证明核心能力的项目，使用 **STAR 法则**展开。

**格式**：

```markdown
### 项目名称

**背景（Situation）**：一句话说明项目背景和挑战

**任务（Task）**：你的角色和职责

**行动（Action）**：

- 你做了什么（技术方案、架构决策、关键实现）
- 解决了什么技术难点

**结果（Result）**：

- 量化的业务成果
- 技术指标提升

**技术栈**：列出项目核心技术
```

**渲染建议**：`行动 / 结果` 最好使用标准 Markdown 列表（标题后空一行再写 `-`），PDF 分块和排版会更稳定。

**选择项目的标准**：
- 技术复杂度高（体现深度）
- 业务影响大（体现价值）
- 你的贡献突出（体现个人能力）
- 与目标岗位相关（体现匹配度）

### 第6章：技术影响力（Technical Influence）

用个人写作、开源贡献、技术演讲来证明技能的深度和热情。

**可包含**：
- 技术博客（平台 + 关注量 + 代表文章）
- 开源项目（项目名 + Star数 + 简介）
- 技术演讲/分享（会议/社区 + 主题）
- 技术书籍/专利
- 社区贡献

**格式示例**：
```markdown
- **技术博客**：掘金/CSDN，累计 10万+ 阅读量
  - 《深入理解JVM垃圾回收机制》（5000+ 阅读）
  - 《从零搭建微服务脚手架》系列（8篇，累计 2万+ 阅读）
- **开源贡献**：参与 Apache Dubbo 社区，贡献 3 个 PR（已合并）
- **技术演讲**：2024 QCon 北京《大模型在金融风控中的工程实践》
```

### 第7章：教育背景（Education）

简洁列出，工作经验丰富时（5年+）此部分从简。

**格式**：
```markdown
**学校名称** | 学历 · 专业 | 毕业年份
```

**可选附加**：
- GPA（如果优秀）
- 获奖情况
- 相关课程/研究方向

## 通用写作规范

### Markdown 格式规范

- 使用清晰的标题层级：`#` 姓名，`##` 章节，`###` 子条目
- 分隔线 `---` 用于分隔主要章节
- 加粗 `**关键词**` 突出核心信息
- 列表 `-` 用于并列内容
- 行内代码 `` ` `` 可用于标注技术术语

### 内容精炼原则

- **控制篇幅**：2-3页为宜，不超过4页
- **删减冗余**：每句话都要有信息量，删掉"负责"、"参与"等弱词
- **聚焦近期**：5年前的经历一笔带过
- **匹配岗位**：根据目标岗位调整侧重点
- **诚实准确**：所有数据可被验证

### 常见问题与修正

| 问题 | 修正方向 |
|------|----------|
| 简历像工作日志 | 提炼成就，删除日常职责描述 |
| 没有量化数据 | 补充数字（规模/效率/指标） |
| 技能堆砌无重点 | 分类+标注熟练度，突出核心技能 |
| 项目描述只有功能 | 用STAR法则重写，强调你的贡献和结果 |
| 缺少一句话定位 | 从核心优势中提炼独特价值主张 |
| 信息布局不合理 | 重要信息上移，F形扫描优化 |
