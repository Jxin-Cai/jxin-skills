# 评审规则

评审时按需选取相关规则，不必逐条检查。

## 通用

- **审查缺失**：没有的东西（缺 .gitignore、缺测试、缺错误处理）往往比写错更严重
- **一致性 > 正确性**：全项目风格统一比局部最优重要
- **考虑项目阶段**：MVP 和生产服务的评审标准不同

## 项目评审

### 结构与配置
- 目录组织反映关注点分离，避免 >4 层嵌套
- 文件命名一致（别混用 camelCase / kebab-case / PascalCase）
- CLAUDE.md：存在、具体可执行、与代码同步、不过度冗长（50-100 行为佳）
- .claude/ 配置：settings.json 权限合理、善用层级 CLAUDE.md（全局→项目→子目录）

### 安全
- 敏感数据不进版本控制（.env / API key / credentials）
- .gitignore 覆盖完整（node_modules、.env、IDE 配置、OS 文件、构建产物）
- 系统边界处校验输入，防注入（SQL / shell / XSS）

### 代码质量
- 函数短小、职责单一
- 错误处理明确，不静默吞异常
- 避免不必要的抽象（YAGNI，三行重复优于错误抽象）
- 命名传达意图，注释解释 Why 不解释 What

### 测试
- 关键路径有覆盖
- 测试行为而非实现
- 测试命名描述场景

### 依赖
- lock 文件提交到版本控制
- 无冗余依赖

### Claude Code 协作
- 代码对 Claude 友好：清晰的类型定义、自解释的函数签名、避免魔法值
- 测试命令单行可执行（不需要交互式输入）
- 善用 Claude Code 生态（hooks、MCP servers、custom slash commands）

## Skill 评审

### 触发设计
- description 同时描述"做什么"和"何时触发"
- 覆盖多种用户表述（中英文、口语、简略）
- 任务型技能收窄触发范围，避免误触发

### 指令质量
- 解释 Why 而非堆砌 MUST/ALWAYS/NEVER
- 步骤按可执行顺序组织
- 避免自相矛盾
- 用示例代替大段描述

### 渐进披露
- SKILL.md < 500 行，详细内容拆到 references/
- references 超 300 行要有目录
- SKILL.md 明确指示何时读取哪个参考文件

### 资源组织
- scripts/ 可独立执行，有 usage 说明
- references/ 内容不过时
- assets/ 只放被引用的文件

### 健壮性
- 处理用户输入不完整的情况
- 不过度拟合特定场景
- 关键决策才等确认，其他用合理默认值推进
