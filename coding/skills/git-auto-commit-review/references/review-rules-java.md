## 并发与线程池（Java）

1. **显式配置线程池（P0）**
   - **原因**：默认线程池和临时 Executor 在生产环境中有风险
   - **反例**：使用 `Executors` 默认配置或无界队列
   - **正例**：设置核心/最大线程数、队列容量和拒绝策略

2. **业务异步任务与 RPC 线程池隔离（P0）**
   - **原因**：共享线程池可能饿死关键线程
   - **反例**：业务任务跑在 RPC 线程池上
   - **正例**：按负载类型使用独立线程池

3. **多线程代码使用线程安全集合（P0）**
   - **原因**：非线程安全的集合在并发下会损坏数据
   - **反例**：多线程共享 `ArrayList` 或 `HashMap`
   - **正例**：使用并发集合或显式加锁

4. **禁止在同一线程池中嵌套提交任务（P0）**
   - **原因**：内层任务提交到运行外层任务的同一线程池会形成循环等待，导致死锁
   - **反例**：外层 `CompletableFuture` 在 `asyncServiceExecutor` 上运行，内层又调用 `CompletableFuture.supplyAsync(..., asyncServiceExecutor)`
   - **正例**：内层任务使用独立的专用线程池；或展平为单层并行执行；或使用 `thenCompose`/`thenApply` 链式调用避免嵌套

5. **始终释放 ThreadLocal 值（P1）**
   - **原因**：线程复用可能导致请求间状态泄露
   - **反例**：调用了 `ThreadLocal.set()` 但没有在 `finally` 中 `remove()`
   - **正例**：在 `finally` 块中清理 ThreadLocal（尤其是线程池场景）

## 幂等与重试（Dubbo）

1. **避免隐式 Dubbo 重试（PS）**
   - **原因**：默认重试可能放大负载和副作用
   - **反例**：依赖 Dubbo 默认重试配置而未审视
   - **正例**：显式配置重试策略并验证幂等性

## RPC 超时（Dubbo/Feign）

1. **按客户端/端点配置 RPC 超时（PS）**
   - **原因**：全局默认超时往往太宽或太紧
   - **反例**：Dubbo/Feign/HTTP 无客户端级超时配置
   - **正例**：设置全局 + 接口级 + 客户端级超时

## 数据库（MyBatis）

1. **MyBatis 参数类型要与索引列类型匹配（P0）**
   - **原因**：类型不匹配导致 MySQL 隐式转换，使索引失效，触发全表扫描
   - **反例**：Mapper 方法的参数用 `String` 类型，但数据库列定义为 `INT`/`BIGINT`
   - **正例**：确保 Java 参数类型与数据库列类型精确匹配（如 `Long` 对应 `BIGINT`，`Integer` 对应 `INT`/`TINYINT`）。参考 mapper XML 中的 `resultMap` 定义来验证列类型

2. **更新操作优先使用主键（P2）**
   - **原因**：非唯一 WHERE 条件可能更新到非预期行
   - **反例**：使用宽泛的非主键条件做更新
   - **正例**：使用主键或唯一约束

3. **明确 DTS 复制延迟的影响（P2）**
   - **原因**：复制延迟下读取到的可能是过期数据
   - **反例**：业务逻辑假设同步立即生效
   - **正例**：接受延迟或提供补偿/修复策略

4. **MyBatis WHERE 条件防御空参数（P1）**
   - **原因**：null 参数可能生成非预期的 SQL
   - **反例**：`WHERE col = #{param}` 但 `param` 可能为 null
   - **正例**：使用 `<if>` 判断和动态 SQL

5. **避免在 MySQL 上做全文检索（P2）**
   - **原因**：LIKE 查询不适用于搜索场景的扩展
   - **反例**：用 MySQL LIKE 做全文或列表搜索
   - **正例**：使用 ES 或专用搜索/缓存索引

## 多租户（Dubbo）

1. **避免 Dubbo 隐式租户传递（P1）**
   - **原因**：RPC 上下文在异步流程中可能破坏隔离
   - **反例**：使用 `RpcContext.setAttachment` 传递租户
   - **正例**：在 RPC 接口中显式传参

2. **使用 Redis 时先加锁再查询（PS）**
   - **原因**：先查后锁可能引入竞态条件
   - **反例**：先检查锁是否存在，再获取锁，再执行业务逻辑
   - **正例**：先获取锁，再查询状态，再执行业务逻辑

## API 契约（Dubbo）

1. **校验跨系统枚举映射（P2）**
   - **原因**：枚举转换可能在服务间漂移
   - **反例**：映射枚举时不处理未知值
   - **正例**：提供兼容性映射和默认值

2. **确保 RPC 泛化调用的类型匹配（P1）**
   - **原因**：类型不匹配导致序列化丢失
   - **反例**：泛化 DTO 字段类型不一致
   - **正例**：保持字段类型对齐；测试序列化

3. **Dubbo API 方法禁止重载（P0）**
   - **原因**：APM 无法区分重载方法的流量
   - **反例**：在 Dubbo API 中使用方法重载
   - **正例**：定义不同名称的独立方法

## 数据校验（Java）

1. **集合不要只做 null 检查（P1）**
   - **原因**：空集合在很多场景下也是无效的
   - **反例**：检查 `list != null` 但忽略了空集合
   - **正例**：需要时同时检查 `isEmpty()`

2. **区分 empty 和 blank（P2）**
   - **原因**：`" "` 在很多场景下应被视为空
   - **反例**：用 `isEmpty()` 校验用户输入
   - **正例**：在空白字符无效的场景使用 `isBlank()`

3. **金额运算使用 BigDecimal（P1）**
   - **原因**：`double/float` 有精度丢失
   - **反例**：用 `double` 存储或计算金额
   - **正例**：使用 `BigDecimal`，明确精度和舍入模式

4. **避免 JSON 反序列化的类型擦除（P2）**
   - **原因**：泛型类型在反序列化时可能丢失字段
   - **反例**：反序列化时不使用 `TypeReference`
   - **正例**：泛型 DTO 使用 `TypeReference`

## 依赖管理（Spring）

1. **显式解决 Spring 循环依赖（P0）**
   - **原因**：循环依赖掩盖了设计问题
   - **反例**：`A -> B -> A` 用 `@Lazy` 敷衍了事
   - **正例**：重构以打破循环；保持依赖无环

2. **禁止在静态初始化器中加载配置（P0）**
   - **原因**：静态初始化块阻塞重载和启动
   - **反例**：`static { loadConfig(); }`
   - **正例**：使用懒加载或托管的配置加载方式

3. **保持环境特定逻辑的一致性（P1）**
   - **原因**：国内/海外分支会随时间漂移
   - **反例**：分支分叉而未使用功能开关
   - **正例**：使用环境标记和一致的代码路径
