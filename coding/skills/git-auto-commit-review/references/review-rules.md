## 规则格式

新增规则时，使用以下模板追加：

1. **规则**：<简短祈使句>
   - **原因**：<一句话>
   - **反例**：<简短示例>
   - **正例**：<简短示例>

## 核心规则

1. **同时审查未暂存和已暂存的 diff**
   - **原因**：最终提交的内容以暂存区为准，不是当前工作区的修改
   - **反例**：只看 `git diff`，遗漏了已暂存的变更
   - **正例**：同时审查 `git diff` 和 `git diff --staged`，核对两者差异

2. **每条建议都要解释原因**
   - **原因**：没有理由的建议难以评估，容易被忽略
   - **反例**："重构这个文件"
   - **正例**："提取辅助函数以减少重复并集中校验逻辑，降低出错面"

3. **区分必须修复和建议改进**
   - **原因**：保持提交节奏的同时持续提升质量
   - **反例**：所有建议混在一起，没有优先级
   - **正例**：分为"必须修复"和"建议改进"两个章节

4. **验证正确性和边界情况**
   - **原因**：大多数回归 bug 来自缺失的边界处理
   - **反例**：假设输入总是非空、非 null
   - **正例**：校验输入；定义空值、null、非法值和边界情况的行为

5. **保持变更小而可审查**
   - **原因**：更小的 diff 降低风险，提升审查质量
   - **反例**：一个 commit 混合了重构 + 行为变更 + 格式调整
   - **正例**：关注点分离，每个 commit 只做一件事

6. **禁止提交密钥或敏感数据**
   - **原因**：泄露的密钥代价高昂且难以补救
   - **反例**：意外提交了 API key、token、凭证或内部 URL
   - **正例**：使用环境变量、密钥管理器、脱敏处理，并在 `.gitignore` 中添加对应规则

7. **保持向后兼容（适用时）**
   - **原因**：破坏性变更必须是有意为之且充分沟通的
   - **反例**：未做废弃策略就重命名公共 API
   - **正例**：先标记废弃，保留兼容层，或使用 API 版本控制

8. **显式处理错误**
   - **原因**：静默失败是最难排查的 bug 来源
   - **反例**：空 catch 块；吞掉错误返回值
   - **正例**：记录/返回有意义的错误信息；带上下文向上传播；补充测试

9. **日志要有目的且安全**
   - **原因**：日志可能泄露数据并增加噪音
   - **反例**：打印含 PII/密钥的完整请求体
   - **正例**：只记录标识符/摘要；脱敏敏感字段；使用合适的日志级别

10. **确认测试覆盖**
    - **原因**：没有测试的变更增加回归风险
    - **反例**：修改了行为但没有测试，也没有手动测试计划
    - **正例**：添加/调整测试，或在报告中记录手动测试方案

11. **标注性能风险**
    - **原因**：小改动可能产生大热点
    - **反例**：每个请求都触发 N+1 查询 / 重复的 JSON 解析
    - **正例**：使用缓存、批量处理、避免重复计算；标注复杂度

12. **保持一致性：命名、风格、模式**
    - **原因**：不一致的代码更难维护
    - **反例**：无理由地引入新模式
    - **正例**：遵循已有约定；偏离时说明理由

13. **避免代码坏味道（参考 Martin Fowler《重构》）**
    - **原因**：代码坏味道暗示更深层的设计问题，增加维护成本和出错风险
    - **反例**：容忍过长方法、过大类、重复代码、依恋情结、数据泥团、基本类型偏执、过长参数列表、发散式变化、霰弹式修改、消息链、中间人、夸夸其谈的未来性
    - **正例**：主动重构：提取方法/类、用值对象替换基本类型、消除重复、封装集合、拆分上帝类、去掉不必要的间接层

14. **生产就绪三支柱：灰度发布、可观测性、应急响应**
    - **原因**：生产变更——尤其是重大业务逻辑修改——必须可控，确保问题能被快速发现并安全回滚
    - **适用场景**：新功能、大型重构、影响关键路径的变更、或触及多个组件/业务流程的修改
    - **反例**：
      - 无灰度：新功能直接全量发布，出问题只能全量代码回滚
      - 无可观测性：受影响路径缺少日志、指标或告警，无法快速定位问题
      - 无应急响应：没有开关、没有降级逻辑、没有快速回滚策略
    - **正例**：
      - 灰度发布：使用功能开关、金丝雀发布或 A/B 实验渐进放量；支持按租户、地区或百分比放量
      - 可观测性：在变更路径上添加遥测日志、业务指标和异常告警；通过仪表盘和告警实现快速发现
      - 应急响应：实现开关以即时禁用新逻辑；为异常场景提供降级逻辑；支持热配置重载而无需重新部署

15. **只检查与项目技术栈相关的风险规则**
    - **原因**：不相关的规则浪费审查时间并产生误报
    - **反例**：审查 Python 或前端项目时检查 Java 特定规则（MyBatis、Dubbo、Spring）
    - **正例**：先识别项目的语言和框架，再选择适用的规则章节

## 风险规则（分类）

### 数据量与载荷

1. **禁止无边界的大载荷（P0）**
   - **原因**：过大的列表或全量历史数据会拖垮延迟和内存
   - **反例**：请求/响应 DTO 携带全量历史，无分页或大小限制
   - **正例**：要求分页/大小限制；默认排除大字段

2. **限制批量任务的数据量（P0）**
   - **原因**：无边界的数据集会产生热点和超时
   - **反例**：批量导入任务接受无限行数
   - **正例**：设定阈值；拆分、分页或分批处理

3. **大批量导出使用流式或分页（P1）**
   - **原因**：不分页的全量导出会耗尽内存
   - **反例**：导出接口一次返回所有数据，无分页参数
   - **正例**：分页/分批导出；使用流式写入

4. **导出查询不要忽略分页参数（P1）**
   - **原因**：接受了分页参数却执行全表查询，隐藏了 OOM 和超时风险
   - **反例**：导出处理器接收分页参数，但调用的是全表扫描查询
   - **正例**：实际应用分页参数，或明确文档说明全量导出并配有保护措施

### 下游负载与调用模式

1. **避免对下游的高 QPS 扇出（P0）**
   - **原因**：无边界的扇出可能超过下游 SLA 并拖垮系统
   - **反例**：紧密循环调用下游，无批量或缓存
   - **正例**：估算 QPS；批量合并请求；按需添加缓存

2. **禁止在嵌套循环中调用下游（P0）**
   - **原因**：嵌套调用以不可预测的方式放大延迟和负载
   - **反例**：每个元素都触发一次 RPC/MQ/缓存调用
   - **正例**：批量下游请求或预加载

### 事务与一致性

1. **显式处理分布式事务（P0）**
   - **原因**：跨系统写入可能导致数据不一致
   - **反例**：先写本地 DB 再调 RPC/MQ，无补偿机制
   - **正例**：使用幂等 + 补偿 + 事件驱动设计

2. **保持事务小且仅包含 DB 操作（PS）**
   - **原因**：事务中包含 RPC/IO 会增加锁持有时间和死锁风险
   - **反例**：事务包裹了 RPC/IO/大循环
   - **正例**：将事务范围最小化到 DB 操作；按需拆分

3. **为下游失败定义回滚/补偿策略（P0）**
   - **原因**：下游操作失败可能留下部分状态
   - **反例**：下游失败无回滚策略
   - **正例**：提供补偿任务或重放机制

4. **异步场景要设计最终一致性（P0）**
   - **原因**：异步工作流需要修复路径
   - **反例**：事件驱动流程无对账或重放机制
   - **正例**：添加补偿、对账和重放支持

5. **防止对归档数据执行操作（P0）**
   - **原因**：操作归档数据可能违反业务规则
   - **反例**：业务操作假设归档数据可修改
   - **正例**：定义归档规则和允许的操作

6. **确保跨系统的状态一致性（P0）**
   - **原因**：上下游状态漂移导致难以排查的缺陷
   - **反例**：状态更新未协调或对账
   - **正例**：使用状态机和对账任务

### 并发与线程安全

1. **确保递归有终止条件和深度限制（P0）**
   - **原因**：无边界递归可能栈溢出或挂起
   - **反例**：递归方法无深度保护
   - **正例**：添加最大深度限制或改为迭代

2. **保护共享可变状态（P0）**
   - **原因**：无保护的并发写入会损坏数据
   - **反例**：共享状态更新时无锁或幂等保护
   - **正例**：使用锁/乐观版本号和可重入检查

### 分布式锁

1. **使用细粒度锁键（P0）**
   - **原因**：粗粒度键降低并发性并产生争用
   - **反例**：锁键缺少业务维度
   - **正例**：在锁键中包含业务标识符

2. **安全释放分布式锁（P0）**
   - **原因**：错误的解锁可能释放他人的锁
   - **反例**：解锁时未校验 token 或非原子操作
   - **正例**：使用 token + 原子释放

3. **定义锁失败的降级策略（PS）**
   - **原因**：锁中间件可能故障
   - **反例**：锁失败无兜底逻辑
   - **正例**：定义降级行为或熔断器

### 幂等与重试

1. **可重试操作必须具备幂等性（P0）**
   - **原因**：重试可能导致副作用重复
   - **反例**：可重试的 API 或 MQ 消费者无去重
   - **正例**：使用幂等键和去重表/缓存

2. **验证下游是否支持幂等（P0）**
   - **原因**：上游重试可能放大下游写入
   - **反例**：调用下游写接口未传幂等键
   - **正例**：传递幂等键并确认下游行为

3. **将重试视为行为变更（P0）**
   - **原因**：重试改变了语义和负载模式
   - **反例**：启用重试却未检查幂等性
   - **正例**：确保方法可重复或幂等

### 超时与异常处理

1. **显式设置网络超时（PS）**
   - **原因**：默认或无限超时可能耗尽线程
   - **反例**：RPC/HTTP 超时未设置或设为 0
   - **正例**：配置合理的超时和异常处理路径

2. **禁止吞掉下游异常（PS）**
   - **原因**：隐藏的失败破坏可观测性和控制流
   - **反例**：捕获后忽略或仅记录下游错误
   - **正例**：向上传播或包装上下文；按需添加降级

3. **对下游错误实施熔断和降级（PS）**
   - **原因**：下游故障可能级联
   - **反例**：下游错误被吞掉或静默忽略
   - **正例**：使用熔断器、降级逻辑和清晰的错误传播

### 消息可靠性

1. **防止生产者消息丢失（PS）**
   - **原因**：未确认的发送可能静默丢消息
   - **反例**：MQ 发送无确认或事务
   - **正例**：使用发送确认；失败时重试或补偿

2. **处理消费者重试和幂等（PS）**
   - **原因**：消费者可能丢失或重复处理
   - **反例**：消费失败无重试或去重
   - **正例**：正确 ACK；使用死信队列和去重

3. **避免 MQ 消费积压（P0）**
   - **原因**：积压延迟关键工作流
   - **反例**：消费者吞吐量低于生产速率
   - **正例**：扩容消费者、限流生产者、或批量处理

### 数据库访问与性能

1. **防止慢 SQL 和全表扫描（PS）**
   - **原因**：缺少索引的查询降低性能
   - **反例**：查询未命中索引或大范围扫描
   - **正例**：使用索引；添加 limit/分页

2. **避免无索引的更新/删除（PS）**
   - **原因**：全表操作会锁表并拖慢数据库
   - **反例**：UPDATE/DELETE 的 WHERE 条件未命中索引
   - **正例**：使用命中索引的 WHERE 条件；分批并限流

3. **防御空查询参数（PS）**
   - **原因**：缺失的参数可能导致全表查询
   - **反例**：WHERE 条件允许 null 或空值却不检查
   - **正例**：校验输入；使用动态 SQL 加空值判断

4. **减少批量操作的锁争用（P0）**
   - **原因**：大批量更新可能阻塞热表
   - **反例**：一个事务中执行大批量 UPDATE/DELETE
   - **正例**：分片批量操作、有序加锁、限流

### 缓存

1. **缓解缓存击穿（PS）**
   - **原因**：缓存未命中可能导致 DB 压力骤增
   - **反例**：缓存 miss 直接穿透到 DB，无保护
   - **正例**：使用互斥锁、热 key 保护或布隆过滤器

### 多租户与上下文传播

1. **每个请求都要校验租户隔离（PS）**
   - **原因**：隐式租户字段可能导致跨租户数据泄露
   - **反例**：仅从上下文读取租户，不做校验
   - **正例**：端到端校验租户；对齐请求和上下文

2. **禁止通过隐式上下文传递租户（P0）**
   - **原因**：隐式上下文可能丢失或被伪造
   - **反例**：租户/仓库仅从上下文推断
   - **正例**：显式传递租户；校验一致性

### API 契约与兼容性

1. **发布要保持向后兼容（P0）**
   - **原因**：不兼容的变更在混合版本环境下会出错
   - **反例**：不做迁移就移除或修改字段类型
   - **正例**：使用双写/兼容层和灰度发布

2. **处理未知枚举值（P2）**
   - **原因**：枚举会演进，老客户端可能遇到新值
   - **反例**：遇到未知枚举值就报错
   - **正例**：添加 `UNKNOWN`/默认分支

3. **返回正确的结果码（P2）**
   - **原因**：错误的结果码误导客户端和重试逻辑
   - **反例**：失败时返回 200，或成功时返回非 200
   - **正例**：对齐业务状态和 HTTP/Result 编码

4. **返回空集合而非 null（P2）**
   - **原因**：null 集合迫使调用方做防御性检查
   - **反例**：列表字段返回 `null`
   - **正例**：返回空数组/空列表

### 数据校验与正确性

1. **校验关键消息字段（P1）**
   - **原因**：缺失字段可能导致消费者或工作流异常
   - **反例**：核心消息字段未做校验
   - **正例**：强制必填字段和默认值规则

2. **避免链式调用导致的空指针（P1）**
   - **原因**：深度链式调用隐藏了空值风险
   - **反例**：`get().get()` 无空值检查
   - **正例**：校验外部响应和集合是否为 null

3. **金额运算使用精确类型（P1）**
   - **原因**：浮点数会丢失精度
   - **反例**：用 `float`/`double` 存储或计算金额
   - **正例**：使用 decimal 类型，明确精度和舍入规则

4. **确保业务逻辑条件严密（P1）**
   - **原因**：过宽的条件扩大了影响范围
   - **反例**：条件匹配了比预期更多的状态
   - **正例**：定义边界条件和异常路径

5. **避免依赖无序数据的顺序（P2）**
   - **原因**：JSON/Set 默认无序
   - **反例**：逻辑依赖 map/JSON 的遍历顺序
   - **正例**：排序数据或使用有序结构

6. **上传文件使用唯一命名（P1）**
   - **原因**：非唯一的文件名会覆盖数据
   - **反例**：文件名仅由日期或业务 ID 构成
   - **正例**：包含唯一 ID 或随机后缀

### 依赖与环境管理

1. **避免环境变量名与保留名冲突（PS）**
   - **原因**：冲突可能遮蔽系统默认值
   - **反例**：环境变量名与系统关键字冲突
   - **正例**：遵循命名约定，避免保留名

2. **优先使用配置而非硬编码常量（P2）**
   - **原因**：硬编码值降低灵活性和安全性
   - **反例**：业务值直接写在代码中
   - **正例**：使用配置文件或集中管理的常量

### 日志与可观测性

1. **避免超大日志（P2）**
   - **原因**：打印完整请求体会泄露数据并增加噪音
   - **反例**：日志记录完整的请求/响应对象
   - **正例**：只记录标识符/摘要；脱敏敏感字段

2. **禁止吞掉异常（PS）**
   - **原因**：静默失败隐藏缺陷并破坏监控
   - **反例**：空 catch 块或仅日志记录的错误处理
   - **正例**：向上传播、告警、或用明确的降级逻辑处理

### 测试纪律

1. **避免 mock 跳过失败路径（P2）**
   - **原因**：强制成功的 mock 隐藏了真实的错误处理
   - **反例**：mock 永远返回成功，异常路径从未被覆盖
   - **正例**：在测试或手动测试方案中包含失败用例
